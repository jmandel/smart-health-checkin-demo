SMART Health Check-in

A browser-based protocol for secure health data sharing
Live Demo GitHub

What is this?

SMART Health Check-in is an open protocol that lets patients share health records from their personal health apps directly with healthcare providers through a browser. It uses OpenID for Verifiable Presentations (OID4VP) with end-to-end encryption, signed request objects, and support for both same-device and cross-device flows.

Same-Device (Patient Portal)

Patient is on their own device. They click a button, a popup opens, they pick their health app, review and share data. The popup returns to the portal page with a response_code to close the loop.

Cross-Device (Front Desk Kiosk)

Staff starts a check-in at a workstation. A QR code appears. The patient scans it on their phone, picks their health app, and shares data. The kiosk receives the encrypted result via long-poll.

Both flows use the same protocol stack, the same shim library, the same picker, the same wallet/source app, and the same verifier backend.

Roles

Verifier The healthcare provider requesting data. Controls the well_known: origin, hosts metadata and signing keys, signs Request Objects, stores encrypted responses. The browser-side shim library (smart-health-checkin) is part of the Verifier's frontend. In the demo: the relay/backend server + the portal or kiosk page.
Picker An optional routing UI where the patient chooses which health app to use. Receives only the minimal bootstrap parameters and forwards them. In the demo: the check-in page.
Wallet / Source App The patient's health app that holds their data. Verifies the signed Request Object, collects consent, encrypts the response, and POSTs it to the Verifier's response endpoint. In the demo: Flexpa.

Protocol Layers

1. Verifier Identity: well_known:

The verifier identifies itself with a custom OID4VP client identifier prefix:

client_id = well_known:https://clinic.example.com

The wallet resolves metadata by fetching /.well-known/openid4vp-client from the verifier origin. This document provides:

  • jwks_uri — signing keys for verifying Request Objects
  • Optional display fields (client_name, logo_uri) — but these must NOT be trusted without an out-of-band allowlist

2. Bootstrap Request

The URL shown to the wallet (via popup or QR code) is intentionally minimal:

client_id=well_known:https://clinic.example.com
request_uri=https://clinic.example.com/oid4vp/requests/abc123
request_uri_method=post

This keeps QR codes small. All real parameters live in the signed Request Object.

3. Signed Request Object

The wallet POSTs to request_uri and receives a signed JWT containing:

  • response_uri — request-specific write endpoint (validated against metadata)
  • client_metadata.jwks — ephemeral encryption key for this transaction
  • dcql_query — what credentials are being requested
  • state, nonce — correlation and freshness

The wallet verifies the JWT signature against keys from jwks_uri before trusting any of these values. Because the Request Object is signed by the Verifier's key, the response_uri is authenticated by the signature itself — no separate metadata validation is needed.

4. Encrypted Response (direct_post.jwt)

The wallet encrypts { vp_token, state } as a JWE using the ephemeral key, then POSTs it to response_uri. The verifier backend stores the opaque ciphertext without being able to read it.

The response endpoint uses first-write-wins: the first POST is authoritative, retries with the same JWE are idempotent, and a different payload is rejected with 409.

5. DCQL Data Profile: smart_artifact

Credentials are requested using DCQL with format smart_artifact. Responses use inline references for deduplication:

{
  "vp_token": {
    "req_insurance": [{
      "artifact_id": "cov_1",
      "type": "fhir_resource",
      "data": { "resourceType": "Coverage", ... }
    }],
    "req_history": [{
      "artifact_ref": "cov_1"
    }]
  }
}

This profile authenticates the request and provides encrypted transport, but does not prove artifact provenance. Unless an artifact carries its own verifiable proof, it should be treated as a transaction-bound submission.

Same-Device Flow

1
Verifier (browser)
Generate keys & init transaction
Ephemeral ECDH key pair generated in browser. POST /oid4vp/same-device/init with the public key, redirect_uri (current page), and dcql_query. Returns transaction_id, read_secret, request_uri.
2
Verifier (browser)
Open popup to picker
Popup URL: checkinBase/?client_id=well_known:...&request_uri=...&request_uri_method=post. Shim starts listening on BroadcastChannel.
3
Picker
User selects health app
Picker forwards the three bootstrap params to the source app, opens it in a new popup, closes itself.
4
Source App
Verify request & collect consent
Fetch /.well-known/openid4vp-client → jwks_uri → POST to request_uri for signed Request Object → verify JWT signature → extract response_uri, dcql_query, ephemeral key. Show consent UI.
5
Source App
Encrypt & POST response
Encrypt { vp_token, state } as JWE → POST to response_uri. Server returns { redirect_uri: "portalPage#response_code=..." }. Source app navigates there.
6
Verifier (browser)
Receive response_code & fetch result
Popup lands on portal page → maybeHandleReturn() broadcasts response_code via BroadcastChannel → shim POSTs { transaction_id, read_secret, response_code } to /oid4vp/same-device/results → decrypts JWE → returns credentials to app.

Cross-Device Flow

1
Verifier (kiosk)
Generate keys & init transaction
Same key generation. POST /oid4vp/cross-device/init (requires staff session cookie). No redirect_uri. Returns same transaction credentials.
2
Verifier (kiosk)
Display QR code
No popup opened. onRequestStart gives the app launch_url. App renders QR code + copy/open link. Shim starts long-polling /oid4vp/cross-device/results.
3
Picker (Patient's Phone)
Patient scans QR & selects app
Same picker, same bootstrap params. Patient picks their health app.
4
Source App (Patient's Phone)
Verify, consent, encrypt, POST
Same verification flow as same-device. After POST, server returns { status: "ok" } (no redirect). Source app shows "Submission complete."
5
Verifier (kiosk)
Long-poll resolves
Long-poll returns { status: "complete", response: "JWE..." }. Shim decrypts with ephemeral private key, validates state, returns credentials to app.
Cross-device init/results require an authenticated staff session (cookie). This prevents an attacker from creating a transaction, showing the QR to a victim, and stealing the decrypted PHI. The standalone relay server disables cross-device by default.

Security Model

Secret Distribution

SecretKnown ByPurpose
transaction_id Requester only Identifies transaction for result retrieval
read_secret Requester only Authenticates result retrieval
write_token Anyone who fetches Request Object One-time write capability for submitting response
response_code Requester (via redirect) Same-device loop-closure proof
request_id / state Public Correlation handle; NOT sufficient to read data
Ephemeral private key Requester browser memory only Decrypts JWE; never leaves the browser

Key Properties

  • E2E Encryption: The relay/verifier backend stores opaque JWE ciphertext. It never possesses the decryption key.
  • First-write-wins: A second submission with different data is rejected (409). Retries with the same JWE are idempotent.
  • Verifier authentication: well_known: + signed Request Objects prove the request is controlled by the domain owner.
  • No blind trust of metadata: Display names and logos from /.well-known/openid4vp-client must NOT be shown as trusted identity without an out-of-band allowlist.
  • Same-device requires response_code: The redirect loop-closure prevents an eavesdropper from racing to fetch the result.
  • Cross-device requires staff session: Prevents anonymous transaction creation and result theft.
  • Artifact authenticity is not guaranteed: Unless an artifact carries its own verifiable proof, it is a transaction-bound submission, not a cryptographically authenticated credential.

Shim API

import { request, maybeHandleReturn } from 'smart-health-checkin';

const result = await request(dcqlQuery, {
  checkinBase: 'https://picker.example.com',
  verifierBase: 'https://clinic.example.com',
  flow: 'same-device',   // or 'cross-device'
  onRequestStart(info) {
    // info.bootstrap  — { client_id, request_uri, request_uri_method }
    // info.launch_url — full URL for popup or QR
    // info.transaction — { transaction_id, request_id, read_secret }
  }
});

// result.credentials['coverage-1'][0] → FHIR Coverage resource

Both flows use the same request() call. The difference is in what the app does with onRequestStart: same-device ignores it (the shim opens the popup), cross-device renders info.launch_url as a QR code.

Demo Architecture

PortalSame-device demo. Calls shim with flow: 'same-device'.
KioskCross-device demo. Staff login, QR code, long-poll. flow: 'cross-device'.
PickerShared routing page. Receives bootstrap params, forwards to source app.
Source (Flexpa)Mock wallet. Fetches metadata, verifies signed Request Object, collects consent, encrypts and POSTs response.

Everything runs on a single port via bun demo/serve-demo.ts. For local development, ./start-local.sh starts multi-origin servers to simulate real cross-origin security.
